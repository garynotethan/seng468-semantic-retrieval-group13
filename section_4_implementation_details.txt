4. Implementation Details

4.1 API Gateway (Flask)
The API Gateway is implemented using Flask (Python 3.12) and serves as the primary entry point for all client interactions. It listens on port 8080 and manages the following core responsibilities:
- Identity Management: User registration and authentication are handled via JWT (JSON Web Tokens) using the flask-jwt-extended library. Passwords are securely hashed using Scrypt (via Werkzeug).
- Document Orchestration: When a document is uploaded, the API performs a multi-step orchestration:
    1. Stream the file to MinIO object storage.
    2. Persist document metadata (UUID, filename, owner, status) in PostgreSQL.
    3. Publish a background processing task to RabbitMQ with the document ID and storage path.
- Search Interface: Exposes a semantic search endpoint that transforms natural language queries into vector embeddings and performs similarity searches against the indexed document chunks.

4.2 Database & Vector Search (PostgreSQL + pgvector)
We utilize PostgreSQL as our primary relational store, enhanced by the pgvector extension to support high-dimensional vector data. 
- Schema Design: The system uses three primary tables:
    - users: Stores account credentials and unique identifiers.
    - documents: Stores top-level document metadata, including processing status ('processing', 'completed', 'failed') and page counts.
    - document_chunks: Stores the individual text segments extracted from documents along with their corresponding 384-dimensional embeddings.
- Scalability & Performance: To ensure low-latency search as the document corpus grows, we implemented an HNSW (Hierarchical Navigable Small World) index on the embedding column of the document_chunks table, optimized for cosine distance search.

4.3 Object Storage (MinIO)
Raw PDF files are stored in MinIO, an S3-compatible object storage server. 
- Multi-tenancy: Files are organized within a single "documents" bucket using a hierarchical pathing strategy: {user_id}/{document_uuid}.pdf. This ensures that user data is logically separated and easily accessible by both the API (for deletions) and the Worker (for processing).
- Resilience: Storage operations are decoupled from the processing logic, allowing the original PDF to remain available even if processing tasks need to be retried.

4.4 Background Worker & Messaging (RabbitMQ)
Asynchronous processing is managed via RabbitMQ, which decouples heavy compute tasks from the request-response cycle of the API.
- Task Queue: The API publishes lightweight JSON messages to a durable "document_processing" queue.
- Worker Pipeline: The background worker consists of a Python-based consumer that executes the following pipeline:
    1. Extraction: PDF files are pulled from MinIO and converted to Markdown using PyMuPDF (fitz) and pymupdf4llm for high-fidelity text extraction.
    2. Chunking: Extracted text is split into chunks of 500 characters with a 100-character overlap to preserve semantic context.
    3. Embeddings: Chunks are converted into vector embeddings using the 'all-MiniLM-L6-v2' SentenceTransformer model. This model provides an optimal balance between inference speed and semantic accuracy.
    4. Indexing: The generated embeddings and text chunks are persisted back to PostgreSQL, and the document status is updated to 'completed'.

4.5 Semantic Search Logic
Search is performed using vector similarity rather than simple keyword matching. When a user submits a query, the API generates a query embedding and executes a SQL query using the cosine distance operator (<=>):
    SELECT text, 1 - (embedding <=> :query_vec) AS score
    FROM document_chunks
    WHERE user_id = :user_id
    ORDER BY embedding <=> :query_vec
    LIMIT 5;
This approach allows the system to retrieve relevant document segments even if the exact query terms do not appear in the text, provided they are semantically similar.
